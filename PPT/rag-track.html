Recommended sequence

How to teach these four projects without losing students

The mistake is to jump directly into app code or LangFlow JSONs. The cleaner path is: first explain retrieval, then show the visual implementation, then show the app architecture, and finally use the embeddings visualizer whenever chunking and similarity need to become concrete.

Projects 12 → 13 → 14 + 15 as explainer
1
Project 12Teach what RAG is, what retrieval fixes, and which architecture fits which QA problem.
2
Project 13Show the same ideas as visual flows so students can inspect each node and prompt.
3
Project 14Show how the theory becomes a runnable product with upload, routing, vector storage, and chat.
4
Project 15Use as a teaching aid when students do not yet understand chunk overlap, embeddings, or similarity.
Project 12

RAG Basics

Project 12 is the theory and reference layer. It explains the full retrieval pipeline and covers the major RAG architecture families before students touch LangFlow or app code.

Folder: Project_12_RAG_Basics

What students get here

  • Clear definition of RAG and why it matters for QA teams with private data.
  • A breakdown of all 10 major RAG architecture patterns used later in the course.
  • Evaluation guidance so students can test retrieval quality instead of trusting answers blindly.

What this project is not

  • It is not the visual builder layer.
  • It is not the production application layer.
  • It should be taught before LangFlow import workflows.

Project 12 map

The 10 RAG patterns covered in the theory track

This is the core classification students should remember. Do not treat RAG as one thing. The value of this project is that it separates retrieval approaches by behavior and use case.

Naive RAG

The simplest retrieve-then-generate pattern. Good as the baseline.

Advanced RAG

Adds query rewriting, chunking strategy, and reranking for better retrieval quality.

Modular RAG

Breaks retrieval into replaceable components and domain-specific branches.

Graph RAG

Uses entities and relationships for multi-hop reasoning and system dependency questions.

Agentic RAG

Uses tools and decision-making to choose actions dynamically.

Self-RAG

Adds self-reflection and quality checks before or after retrieval.

Corrective RAG

Evaluates retrieval quality and falls back to other sources when context is weak.

Hybrid RAG

Combines vector retrieval with lexical search such as BM25.

Multi-Modal RAG

Handles images, tables, charts, and other non-plain-text sources.

Contextual RAG

Adds chunk-level metadata and richer context to improve retrieval relevance.

Project 12 testing

Evaluation is part of the curriculum, not an afterthought

A useful RAG course must teach how to test the system. Project 12 explicitly includes RAG evaluation so students learn to check retrieval quality, answer groundedness, and failure behavior.

What to measure

  • Did the retriever fetch the right chunks?
  • Did the answer stay grounded in retrieved context?
  • Did the system avoid inventing unsupported facts?
  • Did the chosen pattern match the QA use case?

Project 13 operations

How to import and run the LangFlow material

This should be part of the workshop script. Students usually understand the flow visually but fail on setup. This section keeps the operational path explicit.

Import workflow

  1. Run LangFlow locally with langflow run.
  2. Open the app at http://localhost:7860.
  3. Use New Flow → Import and choose the JSON.
  4. Configure provider keys such as OpenAI, Groq, or Cohere.
  5. Test the flow in the playground before teaching from it live.

Project 14

RAG with VIBE Coding

Project 14 architecture

How the modular app works

The code is straightforward: one FastAPI app, one upload route, one chat route, one router module, one ingestion module, and one generation module. This is the best point in the track to connect theory, visual flow thinking, and actual app delivery.

1. Upload

Users upload PDF, CSV, TXT, or Markdown files and assign the content to a domain.

2. Ingest

The ingester extracts text, chunks it, and writes vectors into domain-specific Chroma collections.

3. Route

The router classifies each question into api, ui, or perf.

4. Retrieve

The generator opens the right vector store and retrieves the most relevant chunks.

5. Generate

The final answer model produces a grounded answer from retrieved context.

6. Display

The static dashboard exposes ingestion and chat from the same hosted UI.

router.py
- ChatOpenAI structured output
- returns api | ui | perf
- fallback defaults to api

ingester.py
- extract text
- split into chunks
- add texts to Chroma collections
generator.py
- route question
- get vector store
- retrieve top-k chunks
- prompt answer model
- return domain + answer

Project 15 providers

Embedding providers available in the app

The default classroom mode should usually be the deterministic demo because it removes setup friction. Move to real providers only after students understand the concept.

Deterministic Demo

No API key. Best for immediate classroom explanation and zero setup friction.

Ollama / Nomic

Local and free. Recommended when you want real embeddings without cloud dependence.

OpenAI

Uses text-embedding-3-small for a simple hosted production-like path.

Suggested first workshop setup

  • Start with demo mode.
  • Explain chunk size and overlap first.
  • Show heatmap second.
  • Switch to Ollama or OpenAI only after students understand the visualization.
